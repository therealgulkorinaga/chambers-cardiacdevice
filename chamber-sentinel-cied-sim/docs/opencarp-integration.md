# openCARP Integration Guide

## Overview

The Chamber Sentinel CIED Simulator supports two EGM waveform generation modes:

- **Mode A (Parametric)** — Default. Uses built-in Gaussian waveform templates. Fast and lightweight, suitable for architecture comparison and data-flow testing.
- **Mode B (openCARP)** — Optional. Uses pre-computed waveform templates generated by openCARP's ionic models (ten Tusscher 2006, O'Hara-Rudy 2011, Courtemanche 1998). Produces biophysically accurate intracardiac electrograms that pass expert electrophysiologist review.

Mode B is a **template library** approach: openCARP runs offline to pre-compute waveforms, which are saved as `.npy` files. At simulation runtime, there is **zero openCARP dependency** — the simulator simply loads the pre-computed templates.

## Why openCARP?

We evaluated four established cardiac simulation tools:

| Tool | What it does | Why we chose / didn't choose it |
|------|-------------|-------------------------------|
| **openCARP** | Ionic-model-based cardiac EP simulation | **Chosen.** Fills our biggest gap (Gaussian EGMs → biophysical EGMs). Python CARPutils, Docker support, active community. |
| IDHP Model | Closed-loop heart-pacer interaction | Not chosen for v1. MATLAB-based. Best candidate for v2 closed-loop dynamics. |
| UPenn VHM | Formal verification of pacing modes | Not chosen. Orthogonal to data-flow comparison. Valuable for device safety verification. |
| CVSim | Hemodynamic modeling | Not chosen for v1. Would add BP/CO outputs. Candidate for v2. |

## Quick Start

### Option 1: Generate templates locally (no openCARP needed)

The template generator includes a **synthetic fallback** that produces enhanced waveforms (better than Mode A Gaussians but not ionic-model-based):

```bash
# Generate templates using synthetic fallback
make generate-templates

# Verify templates
make validate-templates

# Run simulation in Mode B
make demo-opencarp
# or
python -m scripts.demo 30 1 --egm-mode opencarp
```

### Option 2: Generate templates with openCARP via Docker (recommended for full fidelity)

```bash
# Pull openCARP Docker image
docker pull opencarp/opencarp:latest

# Generate templates via Docker (2-4 hours)
make generate-templates-docker

# Verify
make validate-templates

# Run simulation in Mode B
make demo-opencarp
```

### Option 3: Native openCARP build on macOS

```bash
# Install dependencies via Homebrew
brew install cmake gcc open-mpi petsc

# Clone and build openCARP
git clone https://git.opencarp.org/openCARP/openCARP.git
cd openCARP && mkdir build && cd build
cmake -DCMAKE_C_COMPILER=gcc-14 -DCMAKE_CXX_COMPILER=g++-14 .. && make -j$(sysctl -n hw.ncpu)
export OPENCARP_HOME=$(pwd)/..
cd /path/to/chamber-sentinel-cied-sim

# Install CARPutils
pip install carputils

# Generate templates natively
make generate-templates
```

## Graceful Fallback

If Mode B is requested but templates are not available, the simulator:
1. Logs a warning: `"openCARP templates not found. Falling back to parametric mode."`
2. Silently falls back to Mode A
3. All other simulation behavior is unaffected

This means the simulator always works, even without openCARP templates.

## Template Library Structure

```
src/generator/cardiac/opencarp/templates/
├── template_catalog.json       # Metadata manifest
├── nsr_atrial.npy             # NSR atrial channel (shape: [100, 1000])
├── nsr_ventricular.npy        # NSR ventricular channel
├── nsr_shock.npy              # NSR shock/far-field channel
├── af_atrial.npy              # Atrial fibrillation atrial channel
├── af_ventricular.npy
├── af_shock.npy
├── vt_monomorphic_atrial.npy
├── ...                        # ~54 files (18 rhythms × 3 channels)
└── .gitkeep
```

Each `.npy` file is a 2D array: `[n_beats, samples_per_beat]` at 1000 Hz source rate. The IonicAdapter resamples to the target device rate (256 Hz) and time-stretches to match the target RR interval at runtime.

## Validation

```bash
make validate-templates
```

Checks:
- All 18 rhythm types present in catalog
- No NaN/Inf values
- Amplitude within physiological bounds per channel
- Beat duration within expected range
- Spectral analysis: dominant frequency matches expected range per rhythm
- Cross-channel consistency

## Limitations

- **No closed-loop pacing**: openCARP generates the cardiac substrate; pacing artifacts are overlaid separately. The heart does not respond to pacing in a biophysically coupled way.
- **Template discretization**: Real openCARP simulations are continuous; we discretize into beat-level templates. Rate-dependent action potential duration changes are approximated via time-stretching.
- **Simplified tissue geometry**: Templates use slab/ring/wedge geometries, not patient-specific anatomical models.
